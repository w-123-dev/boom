# 🎮 炸弹人对战平台 — 项目策划书

**项目名称：** BoomArena（炸弹人竞技场）
**开发周期：** 2026年6月 — 2026年7月
**开发语言：** JavaScript（Node.js + 纯前端）
**项目定位：** 多人实时对战网页游戏

---

## 1. 需求分析

### 1.1 项目背景

2026年世界杯期间，同学群体有大量空闲时间需要线上娱乐。传统的单机游戏无法满足社交需求，而大型联机游戏对硬件要求高。一个轻量级的网页对战游戏平台，无需下载安装、打开浏览器就能和朋友对战，正好填补这个需求。

### 1.2 目标用户

- 高校学生（18-24岁）
- 有短时娱乐需求
- 希望和朋友在线互动
- 电脑配置不高（集显+4GB内存也能流畅运行）

### 2.2 技术选型与理由

| 层级 | 技术 | 版本 | 选型理由 |
|------|------|------|---------|
| **运行时** | Node.js | v24.16.0 | 已有环境，前后端统一语言 |
| **HTTP 框架** | Express | 4.x | 最成熟的 Node.js 框架，生态庞大 |
| **实时通信** | Socket.io | 4.x | 自动降级（WebSocket→轮询），内置房间管理 |
| **数据库** | better-sqlite3 | 11.x | 零配置、同步API性能极高、无需安装服务 |
| **前端** | 原生 HTML+CSS+JS | — | 无需框架编译，直接修改直接生效 |
| **前端 Canvas** | 原生 Canvas API | — | 2D 游戏轻量级首选 |
| **登录认证** | jsonwebtoken | 9.x | 无状态认证，天然适合前后端分离 |

**为什么不用框架：**
- 游戏前端大量使用 Canvas 绘制，React/Vue 在此场景没有优势
- 去掉编译步骤，修改 HTML/JS 直接刷新浏览器就能看到效果，开发效率更高
- 项目总共不超过 10 个页面，不用路由库也完全可以管理

### 2.4 数据流设计

#### 登录注册流程（HTTP）
```
客户端 POST /api/auth/register → 服务端验证 → 写入DB → 返回 JWT Token
客户端 POST /api/auth/login    → 服务端验证 → 返回 JWT Token
客户端 后续请求 Header: Authorization: Bearer <token>
```

#### 游戏流程（WebSocket）
```
① 连接建立：客户端连接 Socket.io，发送 auth token
② 进入大厅：服务端验证 token，加入 lobby room
③ 创建/加入房间：客户端 emit 'create_room' / 'join_room'
④ 准备开始：房主 emit 'start_game'
⑤ 游戏进行：客户端 emit 'player_input' → 服务端处理 → broadcast 状态
⑥ 游戏结束：服务端判定胜负 → 保存结果 → 通知双方
⑦ 返回大厅：客户端 leave room
```

### 2.5 核心游戏循环设计

```
服务端 Game Loop（20次/秒，即 50ms tick）
  │
  ├── 接收玩家输入（方向键、放炸弹）
  ├── 更新玩家位置（碰撞检测）
  ├── 炸弹倒计时（3秒后爆炸）
  ├── 爆炸范围计算 + 破坏软砖
  ├── 道具生成（被炸的砖下）
  ├── 玩家拾取道具
  ├── 玩家死亡判定（被炸到）
  ├── 胜负判定（只剩1人 → 结束）
  │
  └── 广播状态给所有客户端（当前帧的快照）
        ↓
  客户端收到状态 → Canvas 渲染
```

## 3. 游戏设计

### 3.1 炸弹人核心玩法

| 元素 | 说明 |
|------|------|
| **地图** | 13×13 网格（包含硬砖、软砖、空地） |
| **玩家** | 每局 2-4 人，出生在四角 |
| **操作** | 方向键（↑↓←→）移动 + 空格放炸弹 |
| **炸弹** | 放置后 3 秒爆炸，十字范围（初始1格） |
| **软砖** | 可被炸毁，部分下方有道具 |
| **硬砖** | 不可摧毁，构成地图骨架 |
| **胜负** | 最后存活的玩家获胜 |

### 3.2 道具系统

| 图标 | 名称 | 效果 | 出现概率 |
|------|------|------|---------|
| 💣+ | 炸弹数+1 | 同时放置的炸弹数量上限+1 | 25% |
| 🔥 | 爆炸范围+1 | 炸弹爆炸范围扩大1格 | 25% |
| ⚡ | 移速提升 | 移动速度提升20% | 15% |
| 🛡️ | 护盾 | 抵挡一次爆炸伤害 | 10% |
| 👟 | 踢炸弹 | 可以踢动炸弹（将炸弹推走） | 15% |
| 💀 | 骷髅 | 负面效果：移速降低30%（增加随机性） | 10% |

**地图生成规则：**
1. 固定边框：第一行/列和最后一行/列为硬砖
2. 柱子：行列索引均为偶数的位置放硬砖（如 2,2 / 2,4 / 4,2）
3. 出生点：四角各清空 3×3 范围，确保玩家出生有活动空间
4. 剩余格子：约 60% 放软砖，40% 留空
5. 道具：在软砖被炸毁时随机生成（约 40% 概率掉落道具）

### 3.4 碰撞检测规则

```
移动碰撞：
  玩家试图移动 → 检测目标格
    如果是硬砖 → 不允许移动
    如果是软砖 → 不允许移动
    如果是空地 → 允许移动
    如果是炸弹 → 不允许移动（除非有踢炸弹技能）
    如果是其他玩家 → 不允许移动（重叠）

炸弹爆炸：
  炸弹爆炸 → 沿十字方向扩散
    遇到硬砖 → 停止该方向扩散
    遇到软砖 → 摧毁软砖，继续扩散
    遇到玩家 → 玩家死亡（除非有护盾）
    遇炸弹 → 连锁爆炸（炸掉该炸弹）
```

### 4.2 表结构

```sql
-- 用户表
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT NOT NULL UNIQUE,      -- 用户名（登录用）
    password_hash TEXT NOT NULL,         -- bcrypt 哈希
    nickname TEXT NOT NULL,             -- 显示昵称
    avatar TEXT DEFAULT 'default',      -- 头像
    rating INTEGER DEFAULT 1000,        -- 积分
    total_games INTEGER DEFAULT 0,
    wins INTEGER DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    last_login DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 好友表
CREATE TABLE friends (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,           -- 发起方
    friend_id INTEGER NOT NULL,         -- 接收方
    status TEXT CHECK(status IN ('pending', 'accepted')) DEFAULT 'pending',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (friend_id) REFERENCES users(id),
    UNIQUE(user_id, friend_id)
);

-- 对局表
CREATE TABLE matches (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    room_id TEXT NOT NULL,              -- 房间码
    game_type TEXT NOT NULL,            -- 'bomberman' / 'werewolf'
    max_players INTEGER NOT NULL,
    started_at DATETIME,
    ended_at DATETIME,
    duration_secs INTEGER,
    status TEXT CHECK(status IN ('waiting', 'playing', 'finished')) DEFAULT 'waiting'
);

-- 对局参与者表
CREATE TABLE match_players (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    match_id INTEGER NOT NULL,
    user_id INTEGER NOT NULL,
    player_index INTEGER NOT NULL,      -- 玩家序号（1-4）
    is_winner INTEGER DEFAULT 0,        -- 是否获胜
    kills INTEGER DEFAULT 0,            -- 击杀数（炸弹人）
    survived_secs INTEGER DEFAULT 0,    -- 存活时间
    FOREIGN KEY (match_id) REFERENCES matches(id),
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 聊天消息表（保留最近1000条）
CREATE TABLE chat_messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    room_id TEXT NOT NULL,
    user_id INTEGER NOT NULL,
    message TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 索引（确保查询性能）
CREATE INDEX idx_friends_user ON friends(user_id);
CREATE INDEX idx_friends_friend ON friends(friend_id);
CREATE INDEX idx_matches_room ON matches(room_id);
CREATE INDEX idx_match_players_match ON match_players(match_id);
CREATE INDEX idx_match_players_user ON match_players(user_id);
CREATE INDEX idx_chat_room ON chat_messages(room_id, created_at);
```

### 4.3 数据库规模估算

| 表 | 单条大小 | 1000用户估算 | 10000用户估算 |
|----|---------|-------------|--------------|
| users | ~200B | 200KB | 2MB |
| friends | ~50B | 500KB（平均5好友） | 5MB |
| matches | ~100B | 1MB（假设人均10局） | 10MB |
| match_players | ~60B | 3MB（人均4条） | 30MB |
| chat_messages | ~200B | 2MB（人均20条） | 20MB |
| **总计** | | **~5MB** | **~67MB** |
| 索引 + 开销 | | ~2MB | ~20MB |
| **实际占用** | | **< 10MB** | **< 100MB** |

**结论：** 即使 1 万个活跃用户玩了一年，数据库也在 100MB 以内。SQLite 完全胜任。

| 游戏房间（空载） | ~5MB | 无对局时 |
| 每局游戏（4人） | ~200KB | 地图+玩家+炸弹状态 |
| **总计（空载）** | **~60MB** | 服务刚启动 |
| **总计（20局并发）** | **~65MB** | 日常运行 |
| **总计（50局并发）** | **~70MB** | 极限情况 |

### 5.3 部署要求

```
最低配置（课程演示足够了）：
  CPU：    1 核（任何现代 CPU）
  内存：   256MB （实际运行 ~70MB）
  硬盘：   1GB （项目 31MB + 冗余）
  网络：   1Mbps 上行带宽（支持 ~50 人同时游戏）
  系统：   Windows / Linux 均可

推荐配置（流畅运行）：
  CPU：    2 核
  内存：   512MB
  硬盘：   5GB
  网络：   5Mbps 上行带宽（支持 ~200 人）
```

### 5.4 带宽估算

```
每帧数据包大小（50ms 一帧）：
  玩家位置：4方向 × 4byte = 16byte
  炸弹状态：~40byte
  道具状态：~20byte
  每帧总计：~100byte / 玩家

每玩家每秒：100byte × 20帧 = 2KB/s
每局（4人）：2KB/s × 4 = 8KB/s（上行）
20局并发：160KB/s ≈ 1.3Mbps

结论：家用宽带（50Mbps 上行）完全足够承载 50 局并发。
```

## 8. 项目亮点总结（答辩用）

### 8.1 技术亮点

| 序号 | 亮点 | 详细说明 |
|------|------|---------|
| 1 | **服务端权威架构** | 所有游戏逻辑（移动、碰撞、胜负）都在服务端执行，客户端只负责发指令和渲染。防止作弊，逻辑一致性有保证。 |
| 2 | **WebSocket 实时通信** | 使用 Socket.io 实现双向实时通信，自动处理断线重连、心跳检测。 |
| 3 | **插件化的游戏架构** | 平台层和游戏逻辑层分离，新增游戏只需实现固定的接口（init/handleMove/end），不需要修改平台代码。 |
| 4 | **零配置数据库** | 使用 SQLite 替代 MySQL，不需要安装数据库服务，打开即用，适合课程项目和轻量部署。 |
| 5 | **纯前端 Canvas 渲染** | 不依赖任何游戏引擎，原生 Canvas API 实现 60fps 平滑渲染，客户端插值让移动更流畅。 |

### 8.2 与"传统系统"项目的对比优势

| 对比项 | 传统系统（二手/论坛） | 本项目（炸弹人对战） |
|--------|-------------------|------------------|
| 技术深度 | CRUD 为主，缺少实时性挑战 | 实时同步、状态仲裁、碰撞检测 |
| 趣味性 | 演示完就关掉 | 同学真的会玩 |
| 面试价值 | "做过论坛" → 面试官无追问 | "做过实时对战" → 面试官追问细节 |
| 演示效果 | 表格+列表+表单 | 彩色游戏画面 + 实时对战 |
| 代码复用 | 每个功能独立，可复用性低 | 平台+插件架构，可加新游戏 |

### 8.3 可能的答辩问题与回答

**Q1: 为什么要用 SQLite 而不是 MySQL？**
> SQLite 对于课程项目级别的数据量完全够用，而且是零配置的——不需要安装数据库服务、不需要配置用户权限。项目规模（<1万用户）下，SQLite 的读写性能不低于 MySQL。部署时只需要把 .db 文件放服务器上，备份也只需要复制这一个文件。

**Q2: 如何保证游戏公平（防止作弊）？**
> 我们采用服务端权威架构。客户端只负责两件事：1）向服务端发送玩家操作指令；2）接收服务端的游戏状态并渲染。所有游戏逻辑——包括移动碰撞、炸弹爆炸判定、胜负判定——都在服务端执行。客户端无法修改游戏状态，从根本上杜绝了作弊。

**Q3: 网络延迟怎么处理的？**
> 服务端每 50ms 发送一次完整的游戏状态快照（包含所有玩家位置、炸弹状态、道具位置），客户端收到后直接渲染。虽然会有约 50-100ms 的延迟，但对于炸弹人这种走格子的游戏来说，这个延迟对游戏体验的影响极小。未来如果需要，可以加入客户端预测来进一步优化。

**Q4: 玩家中途断线怎么办？**
> 我们实现了断线重连机制：玩家断线后，服务端会保留其游戏状态 30 秒。如果 30 秒内重新连接，玩家会恢复到断线前的位置继续游戏。如果超过 30 秒未重连，该玩家视为淘汰。

| 风险 | 概率 | 影响 | 应对方案 |

| SQLite 并发写入冲突 | 低 | 低 | WAL 模式开启写并发，游戏数据写入延迟可接受 |

## 10. 结语

本项目的核心设计理念是：**做一个真正有人玩的项目，而不是交完作业就删的代码**。

炸弹人对战是经典玩法，在网页上实现后，同学之间打开浏览器就能对战，不需要下载、不需要注册账号（可以用访客模式）、不需要高配置电脑。技术上覆盖了全栈开发的完整链路（前后端交互、实时通信、数据库、游戏引擎），且每一步都有实际的技术挑战。

彩头游戏"谁是卧底"作为大厅里的轻量互动，进一步增加了平台的社交属性。

整个项目开发周期约 15 天，核心功能（对战）在第 6 天即可完成，余下时间用于打磨和扩展，确保交作业时是一个**功能完整、体验流畅、有技术深度**的作品。
